## 四、关键发现

1. **Error% 的真相**：聚合报告的 Error% 不代表 HTTP 500，而是 JMeter 自身 TCP 连接损耗。所有到达服务的请求均返回 `{"success":true, "data":订单ID}`。

2. **Ramp-up 的线性效果**：0s → 5s → 10s，Error% 14.4% → 9.09% → 7.89%，趋势明显。

3. **代码调优有效性**：
   - `INCRBY` → `SET` 根治 Lua 崩溃
   - 拦截器去重减少 33% Redis 开销
   - 连接池扩容消除超时


## 五、结论

通过 9 轮压测调优，秒杀接口错误率从 100% 降至 7.89%。所有到达服务端的请求均处理成功（`success:true`），剩余 Error% 来源于 JMeter 与虚拟机之间的 TCP 连接损耗，而非业务逻辑错误。

系统在虚拟机资源受限环境下表现稳定，性能瓶颈定位在网络层，业务代码层面已无明显瓶颈。
